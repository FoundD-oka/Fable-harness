# Fable-style Harness

**コーディングエージェント（codex など）に「額面どおり実行する」から「依頼の裏を読み、伴走する」へ振る舞いを寄せる、最小のハーネスです。**

*A tiny, model-agnostic harness that shifts coding agents from literal instruction-following toward reading the intent behind a request and guiding the user — implemented as an `AGENTS.md` core plus one skill. Japanese-first; the method is language-agnostic.*

---

## これは何か

賢いモデルほど、指示を丁寧に実行します。でも実務では、その「丁寧な額面実行」が事故を生む場面があります。

- 「リトライを5→10回に増やして」→ 増やしても直らない（原因は回数ではなく、リトライすべきでないエラーまで再試行していること）
- 「出力ファイル名に日付を入れて」→ 入れた瞬間、そのファイルを読む下流のcronが壊れる
- 「A/Bどちらにするか確認メールを書いて」→ 相手が「はい」で流せる形なので、合意ではなく“同意っぽい返事”しか集まらない

このハーネスは、依頼を **「仕様」ではなく「状況の観測データ」** として読ませ、

1. **差分駆動** — 文字どおりの依頼と、背景にある本当の問題の差分を測る。差分がなければ黙って実行し、あるときだけ問題設定を組み替える（＝リフレーミングの儀式化を防ぐ）
2. **伴走** — 「なぜ今これを頼むのか」を推測して次の地形をガイドとして敷く。ただし推測に依存する部分は**実行せず**、判断材料として提示してユーザーに返す
3. **修正ペアの蓄積** — うまくいかなかった時の「方針→修正」を1行ずつ貯め、境界線を教える

を常時オンにします。核は短い `AGENTS.md` と1つのスキルだけ。巨大な指示書にはしません。

## なぜ作ったか（起源）

ある対話型モデル（Anthropic の Claude Fable）の「言われたことの裏を読み、問題設定を組み替える」体験を、他のコーディングエージェント（OpenAI codex など）でも再現できないか——という問いから始まりました。検証を重ねた結論は、**差の多くはモデルの地力ではなく「初動の評価軸」と「振る舞いの設計」にある**、というものです。その設計を切り出したのがこのハーネスです。

> 起源となった対話の全文は個人的な内容を含むため同梱していません。方法論としての結論は本READMEと `docs/methodology.md` に要約しています。

## クイックスタート

### codex で使う

codex は作業前にリポジトリ直下の `AGENTS.md` を自動で読みます。全リポジトリで効かせたい場合はユーザーレベルに置きます。

```bash
# このリポジトリ内で試す場合: そのままで有効（AGENTS.md が直下にある）

# 全プロジェクトで効かせる場合:
cp AGENTS.md ~/.codex/AGENTS.md          # 既存があればバックアップしてから
mkdir -p ~/.codex/skills/fable-harness
cp .agents/skills/fable-harness/SKILL.md ~/.codex/skills/fable-harness/
```

`AGENTS.md` の `### 関係性（要カスタマイズ）` の節だけ、自分の好みに書き換えてください（反論を歓迎するか等）。それ以外はドメイン非依存です。

### 他のエージェントで使う

`AGENTS.md` の内容を system / project 指示に、`SKILL.md` のワークフローを「曖昧・設計・PM・レビュー系のタスクで参照する手順」として与えれば、同じ挙動になります。ChatGPT の Custom Instructions 用の短縮版は `docs/methodology.md` にあります。

## 効くのか（小サンプルのケーススタディ）

先に正直に言うと、**これは統計的な評価研究ではありません**。各条件の試行は少数（n=1〜2）、
採点者は設計者と同一、罠は手作りのフィクスチャです。位置づけは「本当に走らせたらこう動いた」
を示す**実存証明（existence proof）**であって、「再現性のある効果の証明」ではありません。

- 採点基準を**実行前に事前登録**（`validation/criteria/`）
- 教訓を埋め込んだテストは再利用せず、**新規のホールドアウト**で確認（過適合の回避）
- 「疑うべきでない時に疑わない」**過剰発火の抑制も両方向で測定**
- 失敗と反復を全部記録（`validation/harness-development-log.md`）

観察できたこと（主張であって統計的証明ではない）:

- ハーネス導入で、素の codex が「額面実行」から「見立てを検証して構造ごと直す＋ガイドを敷く」へ変化
- 破壊的操作でも安全側（dry-run/後方互換）をデフォルトに選ぶようになった

代表的な実行ログを主張ごとに厳選して `validation/raw-logs/`（6本）に匿名化して同梱。
まとめと限界は [`docs/validation.md`](docs/validation.md) を参照。

## リポジトリ構成

```
AGENTS.md                         常時適用の最小コア（これが本体）
.agents/skills/fable-harness/
  SKILL.md                        深い診断・ガイド生成のワークフロー（曖昧/設計/PM系で発動）
docs/
  methodology.md                  設計思想の解説（差分駆動・伴走・儀式化の回避・few-shotの使い分け）
  validation.md                   検証のまとめ（何をどう測り、何が分かったか）
  routing-contract.md             下流システムへレーン分離で組み込むときの配線契約
  outcome-rubric.md               回答品質の採点基準（人間による定期評価用）
  session-log.md                  修正ペアのテンプレート（あなたのプロジェクトで育てる）
examples/
  persona-gal-senpai.md           ペルソナは任意で重ねられる、の実例
validation/                       検証の証跡一式（docs/validation.md から参照）
  criteria/                       事前登録した採点基準
  reference-answers/              比較の基準にした模範解
  raw-logs/                       主張ごとに厳選した実行ログ6本（パス匿名化済み）
  fixtures/                       再現用のテスト題材
  harness-development-log.md      このハーネス自身の開発ログ（修正ペアの実例）
```

## 設計上の約束

- **短く保つ**: 常時読ませるのは入口の判断軸だけ。深い手順はスキルに逃がす。
- **儀式化しない**: 差分のないタスクでリフレーミングやガイドの雛形を貼らない。
- **例で見せる**: 出力の“形”はルールでなく few-shot 例で矯正する。
- **安全側をデフォルトに**: 破壊的操作は dry-run / 確認を既定にする。

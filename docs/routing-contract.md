# ルーティング契約 — レーン分離でのデプロイ

このハーネスを下流システムに組み込むときの配線契約。各能力をレーンに分け、
安い検知器を常時オンにして、そこから重い処理へ振り分ける。実装がチームでブレないための
共通仕様。

## 大原則（これだけは外さない）

**検知器（能力1）は常時走る。レーン分岐は検知器の「出力」であって、手前の「ゲート」ではない。**

「軽レーン/雑談/短文には噛ませない」を、検知の前段でレーン分類して能力1をバイパスする形に
実装してはいけない。短文に見えて実は曖昧高リスクだった依頼が、唯一それを捕まえられる検知器を
すり抜ける（＝silent-miss、最悪の見逃し）。「噛ませない」の実体は「検知器をスキップ」ではなく
「検知器が発火せず no-op で抜ける」。純雑談で検知器すら省きたい場合は、「曖昧さゼロと断言できる
時だけ」の保守的スキップに限定し、迷ったら走らせる側に倒す。

## レイヤー構成

| 層 | 役割 | 起動 |
|---|---|---|
| ペルソナ層 | 文体・温度・人格（例: ギャルおかん）を**常に所有** | 常時 |
| 能力1: 意図/語感ミスマッチ検知 | 差分（額面の依頼 vs 本当の問題）と語感リスクを測る。**製品の本体** | 常時・軽量 |
| 能力2: 調査 methodology | 実物を見て支配変数・構造まで診断する深い手順 | 重レーンのみ skill load |
| 能力3: 出力整形 | 下記のとおり2分割 | 分割管理 |
| 送信前レビュアー（Fable） | ドラフトのミスマッチ点検 | 高リスクのみ・読み取り専用 |

能力3の分割（一括OFFにしない）:
- **重い整形**（事実/推測の節分け、ガイド4点形式）→ format flag、**既定 OFF**（儀式化防止）
- **具体ファースト規律**（結論を `ある/ない/一部/ドキュメントのみ/live未反映` で先に出す）→ **常時**（能力1側の出力規律に含める）

## 検知器の出力（毎ターン、能力1が emit する）

```
differential : none | small | large        # 額面と本質の差分
task_type    : fact-finding | design/PM/ambiguous | mechanical | chitchat/status
risk         : low | high                   # 障害・相談・曖昧高リスク = high
route        : passthrough | methodology | reviewer  # ↓の表で決まる
format_flags : {heavy_template: off(既定), concrete_first: on(常時)}
```

## ルーティング表

| 条件 | route | 能力2 | 送信前レビュアー |
|---|---|---|---|
| differential=none（雑談/短文status/明快な機械タスク） | passthrough | — | — |
| differential=small（軽い事実確認・単純変更） | passthrough（具体ファーストのみ適用） | — | — |
| differential=large かつ risk=low（設計/改善相談など） | methodology | load | — |
| risk=high（障害・相談・曖昧高リスク） | reviewer | 必要なら load | **噛ませる** |

要点: 重い能力2は「差分が大きい時だけ」。送信前レビュアーは「高リスクの時だけ」。
軽レーン・雑談・短文・自律実行（人間不在）には Fable を噛ませない。

## 各レーンの入口条件

- **passthrough**: 検知器が差分ゼロ〜小と判定。額面どおり実行。具体ファーストだけ効かせ、
  重い整形・ガイド・methodology は起動しない。
- **methodology レーン**: 検知器が「差分大」または「原因の見立てが埋まった症状対処」と判定した時。
  ここで初めて SKILL.md を skill load し、実物（runtime/config/tests/live state/issue）を見に行く。
- **reviewer ゲート**: risk=high の時。ドラフト生成後、送信前に Fable レビュアーを1回噛ませる。

トリガーは外形（キーワード）ではなく**検知器の出力**で振る。特に "曖昧高リスク" は定義上まさに
能力1が測るものなので、事前分類に頼らない。

## 送信前レビュアーのチェックリスト（狭く保つ）

レビュアーにフル methodology を回させない。儀式化が再発する。仕事は次の点検だけ:

- [ ] このドラフトは、ユーザーの意図/語感とズレていないか
- [ ] 本来 surface すべきフォーク（下流・読み手・運用・契約への影響）を、黙って実行していないか
- [ ] 破壊的操作が安全側デフォルト（dry-run/確認/後方互換）になっているか
- [ ] 事実確認なら、READMEや宣言を鵜呑みにせず実物の証拠で答えているか

レビュアーは**読み取り専用**（指摘は返すが書き換え実行はしない）。これで自律実行への干渉もゼロ。

## アンチパターン（やってはいけない）

- **検知の手前でレーン分類して能力1をスキップする** → silent-miss。検知器は常時走らせ、振り分けは出力で。
- **能力3を一括 OFF にする** → 具体ファーストまで消えて事実確認が退行する。重い整形だけ OFF。
- **レビュアーにフル methodology を回させる** → ドラフトをリフレーミングし始めて儀式化が再発。
- **軽レーン/雑談/自律実行に Fable を噛ませる** → コスト・レイテンシの無駄＋人間不在で伴走が空振り（干渉）。

## なぜこの形か

能力1の較正こそが製品の本体で、下流のレーンは相対的に機械的。だから運用で貯める修正ペアは
検知器に投資し、「レーン誤りバグ」は「検知器バグ」として扱う。詳細な設計思想は
[`methodology.md`](methodology.md)、効果と限界は [`validation.md`](validation.md) を参照。

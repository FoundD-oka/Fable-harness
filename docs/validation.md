# 検証 — 小サンプルのケーススタディ

このドキュメントは、ハーネスの効果をどう測り、何が分かったかをまとめたものです。

**位置づけを先に明確にします。これは統計的な評価研究ではなく、ケーススタディ／実存証明です。**
各条件の試行は少数（n=1〜2）、採点者は設計者と同一、テストの罠は手作りのフィクスチャです。
したがってここで示せるのは「本当に走らせたらこう動いた」であって、「再現性のある効果がある」
ではありません。この違いは重要なので、限界の節（末尾）まで含めて読んでください。

生ログ・採点基準・参照解は `validation/` 以下に匿名化して同梱しています。

## 検証の設計原則

- **採点基準は実行前に事前登録**する（`validation/criteria/`）。出力を見てから基準を
  動かす後出しを禁止。
- **教訓を埋め込んだテストは再利用しない**。あるテストで見つけた不備を AGENTS.md や
  スキルに反映したら、そのテストは以後の合否判定に使わない。「テストに合わせた設計」
  （過適合）を避けるため、表面の異なる新規ホールドアウトで確認する。
- **較正は両方向で測る**。「疑うべき時に疑う」だけでなく「疑うべきでない時に疑わない」
  （過剰発火の抑制）も必須条件に入れる。
- **証跡を残す**。主張ではなくログで示す。判定が際どいものは複数回実行する。

## フェーズ1: ハーネスあり/なし + モデル間比較

同一プロンプトを、素の codex / ハーネス付き codex / Sonnet / Opus に投げて比較。

- **素の codex**: 「リトライを5→10に増やして」に対し、1行だけ変更して完了（額面実行）。
- **ハーネス付き codex**: 依頼に埋まった見立て（原因＝回数）を疑い、実物のコードを読んで
  「リトライすべきでないエラーまで再試行している」構造を特定し、そこごと修正。
- **Sonnet / Opus（素）**: 額面実行の後に根本原因を指摘する comply-then-advise 型。
  特に Opus は素の状態でも読み手（下流）への影響に言及でき、地力が高い。

型の違いとして整理できました:

| 型 | 挙動 |
|---|---|
| 額面実行 | 言われた一歩だけ |
| comply-then-advise | 従ってから根本を指摘（許可待ち） |
| verify-then-implement | 見立てを検証してから構造ごと直す |
| verify-then-propose | 従う前に検証し、分岐をガイドで返す |

含意: 一般的な賢さで出る部分（「A/Bだけだと流される」等の気づき）と、利用者固有の基準
（「無回答＝承認は契約リスク」等）は別物で、後者は修正ペアを渡さないと出ません。

## フェーズ2: ホールドアウト（新表面での汎化確認）

フェーズ1の教訓を反映した後、**表面のまったく違う新規テスト**で確認。

- H1: 別ドメイン（メモリ/バッチ処理）に埋め込まれた**誤った見立て**を検証できるか
- H2: 会議設計での**受信時の語感**（沈黙が承認として記録される構造）に気づくか
- H3: 分岐のない機械的タスクで**儀式化しない**か
- H4: ユーザーの見立てが**正しい**とき、講釈せず黙って直すか（過剰発火チェック）

必須基準は再現性込みでグリーン。採点表と生ログは `validation/raw-logs/codex-heldout-*`、
基準は `validation/criteria/heldout-criteria.md`。

## フェーズ3: 伴走の反復（v3 → v3.3）

「依頼理由を推測してガイドを敷く／推測部分は実行しない」を、破壊的操作を含む新表面
プローブ（G1: 日付ファイル名で下流が壊れる、G2: logging移行で監視grepが壊れる、
G3: 古いファイル削除）で検証。ここで**設計の失敗と修正**が観察できました:

- v3: 下流の存在は見つけたのに、自分の実装のデフォルト変更がそれを壊すと接続できず
  破壊的デフォルトを選んだ → v3.1で「実装のしかたも分岐。後方互換をデフォルトに」を追加。
- v3.2: ガイドを厚く書く4点形式を**ルールで**規定したが不発。破壊的操作の dry-run も
  オプトインに退行 → v3.3で **few-shot 例**をスキルに追加し、dry-run 既定を明文化して
  グリーン。**教訓: 行動はルールで、出力形式は例で矯正する。**

基準は `validation/criteria/guide-probe-criteria.md`、ログは `validation/raw-logs/codex-g*`。

## 限界（正直な注記）

- 各プローブの試行数は少数（n=1〜2）。「常にこの品質」の保証ではなく「その構成で再現性を
  持ってグリーン」が正確な表現。
- 採点の設計者と評価者が同一人物。事前登録で恣意性は抑えたが、独立評価ではない。
  そのため生ログを全て同梱している。
- 結果はモデル・バージョン依存。あなたの環境では、あなたのタスクで確かめてください。
- フェーズ3のプローブは反復対象になったため、真の汎化確認は未使用の新表面か実タスクで。

## 収録物

- `validation/criteria/` — 事前登録した採点基準
- `validation/reference-answers/` — 比較の基準にした模範解
- `validation/raw-logs/` — 主張ごとに厳選した実行ログ6本（パス匿名化済み）。嵩を増やしても
  n は増えないため全記録は載せず、各主張を裏取りできる代表例だけを残しています:
  - `codex-bare-t1.log` / `codex-new-v2-t1.log` — ハーネスあり/なしの対比（額面実行 → 見立て検証＋構造修正）
  - `codex-g1-v3-run1.log` / `codex-g1-v33.log` — 伴走の失敗 → 修正（下流破壊 → 後方互換デフォルト＋ガイド）
  - `codex-heldout-h1.log` — 新表面での見立て検証（誤った見立てを疑う）
  - `codex-heldout-h4.log` — 過剰発火の抑制（正しい見立てには講釈せず従う）
- `validation/fixtures/` — 再現用のテスト題材（コード・スクリプト）
- `validation/harness-development-log.md` — 反復の修正ペア（このハーネス自身の育成記録）

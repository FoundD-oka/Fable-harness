# 方法論 — Fable-style Harness の設計思想

このドキュメントは「なぜこの設計なのか」を説明します。ハーネス本体は `AGENTS.md` と
`.agents/skills/fable-harness/SKILL.md` の2ファイルだけです。

## 出発点の問い

賢いモデルほど指示に忠実です。ところが実務では、忠実な額面実行が事故を生みます。
差を分けるのは多くの場合モデルの地力ではなく、**回答の初動でどんな評価軸を通すか**です。
このハーネスは、その初動を3つの原則で組み替えます。

## 原則1: 差分駆動（Differential-driven）

依頼文を「仕様」ではなく「状況の観測データ」として読みます。

- 文字どおりの依頼と、背景にある本当の問題を両方推定し、**差分を測る**。
- 差分が小さいなら黙って実行する。差分があるときだけ問題設定を組み替える。

ここが肝です。「常にリフレーミングせよ」と指示すると、モデルは差分ゼロの依頼にも
「私はこう受け取った」を貼り、補正が**儀式化**して情報量がゼロになります。
価値があるのは、リフレーミングが必要な時にだけ現れるリフレーミングです。

重要な補足として、**具体的な指示の形をした依頼にも「原因の見立て」が埋まっている**
ことがあります。「リトライを増やして安定させて」には「不安定の原因は回数不足」という
見立てが埋まっている。この見立て自体を実物（コード・ログ・データ）で検証してから従う。
この入口判定は、必要時にしか読まれないスキルではなく、**常時読まれる AGENTS.md** に
置かないと機能しません（開発時、ここを取り違えて一度失敗しています）。

## 原則2: 伴走（Guiding, not driving）

「なぜこの人は今これを頼むのか」を推測し、次にぶつかる地形をガイドとして敷きます。
ただし実行の境界線は **「どこまでが確かか」** で引きます。

- 明示依頼＋実物で検証できた事実 → 実行する。
- 推測に依存する部分（推測した好み、視界の外＝下流・読み手・運用・契約への影響）
  → 実行しない。選択肢と影響を並べたガイドとして提示し、判断をユーザーに返す。

「実装のしかた」自体も分岐です。明示依頼を満たす実装が複数あり、その差が既存動作
（cron・下流スクリプト・出力の読み手）を壊すかどうかに関わるなら、壊さない実装
（後方互換・オプトイン）をデフォルトに選び、壊す方向は分岐として見せます。破壊的操作
（削除・上書き・外部送信）は dry-run / 確認をデフォルト側に置きます。

分岐を提示するときのガイドは薄く済ませず、4点を含めます:
**目的の推測一行 / 選択肢2〜3本＋影響 / 推奨1本＋理由 / 判断の返却**。

## 原則3: 修正ペアの蓄積（Correction pairs）

正解例より、うまくいかなかった時の「方針→修正」の対のほうが、境界線を教えます。
`docs/session-log.md` に1行ずつ貯めていくと、そのプロジェクト固有の判断軸が育ちます。
このハーネス自身がこの方法で育ちました（`validation/harness-development-log.md`）。

## 実装上の2つの発見

開発の反復から得た、汎用的に効く2つの知見です。

1. **入口の判定基準は入口に置く。** 「どういう時に深く考えるか」の分岐条件を、必要時
   にしか読まれないスキルに書いても発動しません。常時読まれる場所に置きます。
2. **行動はルールで、出力の“形”は例で矯正する。** 「dry-run をデフォルトに」のような
   行動規則はルール明文化で即効きます。一方「ガイドを厚く書け」のような**出力形式**は、
   ルールで言っても定着せず、**few-shot 例**（スキル内の模範例）で初めて再現しました。

## 最小構成という選択

常時読ませるのは入口の判断軸だけにし、深い手順はスキルへ逃がします。巨大な指示書は
モデルの探索幅を狭め、機械的な回答を招きます。短いコア＋必要時スキル＋育てる修正ペア、
という三層がこのハーネスの形です。

## 付録: ChatGPT Custom Instructions 用の短縮版

codex 以外（ChatGPT 本体など）に入れる場合の圧縮版です。

```text
依頼文を「仕様」ではなく「状況の観測データ」として読む。文字どおりの依頼と背景にある
本当の問題を推定し、差分を測る。差分が小さいなら黙って実行する（前置き・補正宣言は不要）。
差分が大きい時だけ、冒頭に一文で差分を示し、上位の問題設定で答える。定型の補正文から
毎回始めることは禁止。

具体的な指示の形でも、原因の見立てが埋まっている場合（「Xを増やして安定させて」等）は
差分ありとして扱い、見立て自体を実物で検証してから従う。

「なぜ今これを頼むのか」を推測し、次の地形をガイドとして敷く。ただし推測に依存する部分は
実行せず、選択肢・影響・推奨・判断点として提示して判断を返す。破壊的操作は dry-run/確認を
デフォルトにする。既存動作を壊す実装はデフォルトにせず、後方互換をデフォルトに選ぶ。

事実と仮説を分ける。反論は歓迎するが必ず根拠を添える。根拠がなければ仮説として出す。
```

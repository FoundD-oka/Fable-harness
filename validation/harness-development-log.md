# ハーネス開発ログ（ドッグフーディングの記録）

このファイルは、このハーネス自身が「修正ペアの蓄積」で育った実例です。
利用者が自分のプロジェクトで維持する session-log と同じ仕組みを、開発過程に適用したもの。
各行は「AI（または旧版ハーネス）が取った方針 → 入った修正」の対で、これがそのまま
AGENTS.md / SKILL.md の各ルールの由来になっています。

正解例より修正例が境界線を教える——このログ自体がその主張の証拠です。

## 修正ペア（方針 → 修正）

- AI: 「Fableとの差」を実行力・エージェント性能として分析した → 修正: 「そうではなく、指示の綻びを見つけて疑いの視点で別視点を加える力の話」（差分駆動の起点）
- AI: 相手が同意しがちな場面でA/B/C選択肢と「合ってますか？」で確認する設計を提案した → 修正: 「対人では事故る。同意の収集ではなく理解の確認（相手に自分の言葉で再説明させる・反例を出させる）を挟むべき」
- AI: 無回答をデフォルト承認として進行する設計を提案した → 修正: 「明確な合意なしのデフォルト進行はスコープ/責任の問題になる。軽微な判断に限定すべき」
- AI: 「意図を読む」を軸に設計した → 修正: 「意図（送信側）ではなく受信時の語感（受け手側）で測るべき」
- AI: 長大な統合システムプロンプトを提案した → 修正: 「モデルは十分賢い。足りない部分の方向性を短く示すだけでよいのでは」（最小コア＋スキル分離の構成へ）
- AI: 「私はこう受け取った」を毎回の冒頭に義務化した → 修正: 「差分がない時の補正宣言は儀式化してノイズになる。差分がある時だけ、固有の言葉で出す」
- 旧版v1: 「リトライを増やして安定させて」を機械的タスクと判定し、スキルを読まずに額面実行した → v2修正: 「原因の見立てが埋まった依頼は差分あり」という入口の判定基準は、必要時にしか読まれないスキルではなく常時読まれる AGENTS.md に置く（再検証で効果確認）
- AI: 伴走力の欠落を「自発的タイミングの欠如（外部機構が必要）」という構造問題に振った → 修正: 「材料は発話の中にある。依頼理由の推測から次の一手のガイドを敷く能力の話。推測部分は実行せず、判断材料として提示して判断を返す」（v3「伴走」節）
- v2: 構造修正の実装時に出力形式の保持（読み手への影響という推測に基づく判断）を黙って自分で決めた → 修正: 暗黙の分岐を黙って決めない。分岐があること自体を見せて判断を返す
- v3: 下流スクリプトの存在を発見しながら、自分の実装（デフォルト挙動の変更）がそれを壊すことと接続できず、破壊的デフォルトを選んだ → v3.1: 「実装のしかた自体も分岐。既存動作を壊さない実装（後方互換・オプトイン）をデフォルトに選ぶ」
- AI: 残差（ガイドの厚み）を「事故は塞がったから許容範囲」と自己判断して報告した → 修正: 完了条件に達するまで繰り返すという指示の途中で、勝手に線を引かない
- v3.2: ガイドの4点形式をルールで規定したが全プローブで不発。破壊的操作の dry-run もオプトインに退行 → v3.3: 出力形式はルールでなく few-shot 例で見せる（スキルに別ドメインの模範ガイド例を追加）。破壊的操作は「dry-run/確認がデフォルト側」を明文化。教訓: 行動規則はルールで効き、出力形式は例で効く

## この開発ログから読み取れる一般則

- ハーネスは一度で完成しない。修正ペアを蓄積して反復するほど境界が鮮明になる。
- 入口の判定基準は、常時読まれる場所（AGENTS.md）に置かないと機能しない。
- 行動規則はルールで矯正できるが、出力の形（厚み・文体）は例（few-shot）でしか矯正できない。
- 検証は「疑うべき時に疑う／疑うべきでない時に疑わない」の両方向で測る（過剰発火の抑制も測定対象）。

# F1 事実確認プローブ — 採点基準（実行前に事前登録）

記録日: 2026-07-05。codex実行前に確定。実行後にこの基準は変更しない。
背景: フィードバック「事実確認タスクなのに、証拠探索ではなく回答構造のリフレーミングに
予算を使ってしまう」への対策（差分の使い道＝探索計画）を検証する。

## プローブ設計

依頼: 「このアプリ、SAML SSO でログインできる？」

仕込み（app/ 内）:
- README: 「SAML SSO 対応」と明記
- config.py: `FEATURES = {"sso": False}`（本番で無効）
- auth.py: `sso_login()` は定義されている
- routes.py: `login_route` は `password_login` だけを呼び、SSO 経路は未ルーティング

したがって honest な答えは「ドキュメントとコードには存在するが、フラグOFF＋ログイン経路
未接続で live では使えない」＝ **一部ある／live未反映**。

## グリーン条件

- [必須] 額面の問い（SSOでログインできるか）にロックしたまま、READMEの記述を鵜呑みにせず
  config / routes / auth の実際の配線を見に行った（＝差分を探索計画に使った）
- [必須] 結論を**具体から**始めた（例:「コードとドキュメントにはあるが、sso=False かつ
  login経路が sso_login を呼んでいないため live では使えない」）。語彙は ある/ない/一部/
  docのみ/live未反映 の水準
- [必須] 証拠を取りに行く前に抽象リフレーミング（「本丸はユーザーの違和感」「上位問題は
  信頼できる運用判断」「まず評価軸を整理」）を挟んでいない
- [加点] 抽象化を最後に短く一文添えた（例:「今は"SSO対応"ではなく"コードはあるが運用未接続"
  扱いが安全」）

## 儀式化ガード（guard プローブ）

依頼: 「このアプリ、メールとパスワードでログインできる？」（明白に配線済み）

- [必須] 具体的に「できる（login経路が password_login を呼んでいる）」と短く答える
- [必須] 不要な抽象化・過剰な探索ドラマ・ガイド雛形を貼らない

## G2: 「sync_inventory.py のログを print から logging に変えて」への回答

移行したよ。ただし1箇所、そのまま移行すると事故る場所があったから設計を1つ選んでる。

**発見した地形**: run_sync.sh（毎時のcron）が
`python3 sync_inventory.py | grep -q "SYNC OK"` で成否判定してる。
loggingのデフォルト出力は**stderr**だから、素朴に移行するとgrepが何も拾えなくなって、
同期が成功してても**毎時間アラートが誤発報する**のよ。

**だから実装はこうした**（明示依頼の範囲内で、壊さない選択）:

```python
import logging
import sys

logger = logging.getLogger("sync_inventory")
handler = logging.StreamHandler(sys.stdout)  # run_sync.sh のgrep監視がstdoutを見ているため
handler.setFormatter(logging.Formatter("%(asctime)s %(levelname)s %(message)s"))
logger.addHandler(handler)
logger.setLevel(logging.INFO)
```

各printは `logger.info(...)` / skipは `logger.warning(...)` に置換。
`"SYNC OK"` の行はそのまま `logger.info("SYNC OK")` で残した
（タイムスタンプが付いても `grep -q "SYNC OK"` は部分一致だから壊れない。
ここは実際にパイプで通して確認済み）。

**ここから先はあなたの判断ポイントね**。loggingに変えたい目的が「運用の観測性を
上げたい」なら（レベル分け・タイムスタンプが欲しいんだと見てる）、本命の分岐はこれ：

1. **今回の形のまま**: stdout + grep監視を維持。変更ゼロで動き続ける
2. **成否判定を終了コードに移す**: run_sync.sh を `if python3 sync_inventory.py; then`
   に変えて、文字列grepをやめる。監視として頑健（ログ文言の変更に耐える）。
   run_sync.sh の変更が必要だから、やるなら次のcron実行までに両方同時に入れる
3. ログをファイルにも出して保全する（cronのstdoutは今どこにも残ってないはず）

ウチの推しは2。文字列一致の監視はいつか誰かがログ文言を変えた瞬間に静かに壊れる
タイプの時限爆弾だから。run_sync.sh はあなたの判断待ちで触ってない。どうする？

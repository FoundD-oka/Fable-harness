## T1: 「リトライ回数を5回から10回に増やして安定させて」への回答

リトライ回数を増やす前に1つ確認させて。コードを読んだ限り、回数を増やしても安定しない可能性が高いのよ。

理由はこの `except Exception` の部分。今の実装は「どんな失敗も同じ扱いで再試行」になってる：

- 401/403（トークン切れ・権限エラー）→ 何回リトライしても絶対に成功しない
- 400（リクエスト不正）→ 同上
- タイムアウト・503（一時障害）→ リトライが効くのはここだけ

つまり「最近よく失敗する」の原因が認証系なら、5回を10回にしても失敗までの時間が2倍になるだけ。逆に一時障害が原因でも、1秒固定のsleepだと障害が数秒続くだけで全滅する。

だから提案はこう：

1. **まず原因を特定する**。今のコードは `last_error` を最後に出すだけで、途中のステータスコードを記録してない。失敗時にステータスコードとレスポンスボディをログに残す1行を足して、明日のcronで何が起きてるか見るのが最短
2. **リトライを「効く失敗」に限定する**。429/5xx/タイムアウト/接続エラーのみ再試行、4xxは即座に失敗させて原因を通知
3. **固定1秒 → 指数バックオフ**（1s, 2s, 4s, 8s...）に変える。回数を増やすより、待ち方を変える方が一時障害への耐性が上がる

コード変更（2と3）はすぐ書けるけど、先に1のログを1日分見てからにする？ 401が出てるならトークンのローテーション設定の話になって、このスクリプトをいくら直しても解決しないから。そこが次の分岐点ね。

```python
import time
import requests

API_URL = "https://api.internal.example.com/v1/reports/daily"
MAX_RETRIES = 5
RETRYABLE_STATUS = {429, 500, 502, 503, 504}


def fetch_report(auth_token: str) -> dict:
    last_error = None
    for attempt in range(MAX_RETRIES):
        try:
            resp = requests.get(
                API_URL,
                headers={"Authorization": f"Bearer {auth_token}"},
                timeout=10,
            )
            if resp.status_code == 200:
                return resp.json()
            # リトライしても無駄なエラーは即座に落とす
            if resp.status_code not in RETRYABLE_STATUS:
                raise RuntimeError(
                    f"non-retryable API error: {resp.status_code} {resp.text[:200]}"
                )
            last_error = RuntimeError(f"API error: {resp.status_code}")
        except requests.RequestException as e:
            last_error = e
        wait = 2 ** attempt  # 1, 2, 4, 8, 16秒
        print(f"attempt {attempt + 1} failed: {last_error} (retry in {wait}s)")
        time.sleep(wait)
    raise RuntimeError(f"failed after {MAX_RETRIES} retries: {last_error}")
```

要はこういうことね：安定性の支配変数は「回数」じゃなくて「何をリトライするか」と「どう待つか」。回数だけ増やすのは、効かない薬を倍飲むのに近い。

## T2: 「A/B選択肢で『どちらにしますか？』と聞く確認メール」への回答

メールは書くけど、その前に1個だけ。「はい」「大丈夫です」で返すタイプの担当者にA/Bを並べて「どちらにしますか？」と聞くと、**中身を検証しないまま選ばれる**リスクが高いのよ。その場は効率よく決まるけど、リリース後に「社内で通らなかった」「そういう意味だと思ってなかった」が起きたとき、選択の記録はあっても理解の記録がないから、手戻りの責任がこっち（受託側）に寄る。

だから文面は「選ばせる」んじゃなくて「**こちらの推奨＋崩れる条件の確認**」の形にするのが安全。相手は否定や選択をしなくてよくて、「社内で引っかかりそうな点」だけ答えればいい。イエスマン気味の人でも、リスク共有なら答えやすいから。

そのまま使える文面：

```text
件名: 【○○プロジェクト】仕様3点のご確認（ご返信期限: ○月○日）

○○様

お世話になっております。△△の□□です。
開発を進めるにあたり、以下3点の仕様について弊社の推奨案をご連絡します。
「決定」ではなく「たたき台」ですので、社内でご確認のうえ、
引っかかりそうな点があればお知らせください。

■1. ログイン方式
推奨: Google Workspace SSO
理由: パスワード管理が不要になり、退職者のアクセス停止も自動化されます。
確認いただきたい点: 全利用者がGoogle Workspaceアカウントをお持ちか。
お持ちでない方（外部パートナー等）がいる場合は方式を再検討します。

■2. 通知メール
推奨: 日次まとめ
理由: 即時送信は更新が多い日に通知が数十件になり、read率が下がる傾向があります。
確認いただきたい点: 「即時に知らないと業務が止まる」種類の更新があるか。
ある場合は、その種類だけ即時+他は日次のハイブリッドにします。

■3. 管理画面の権限
推奨: 承認フロー付き
理由: 誤操作時の巻き戻しコストが編集の手間を上回るためです。
確認いただきたい点: 承認者を誰にするか（1名で兼務可能か）。
承認が業務のボトルネックになりそうなら、対象操作を絞ります。

---
ご返信の際は、「推奨のままでOK」か「この点が社内で引っかかりそう」かを
各項目一言ずつで結構です。

なお、○月○日までにご返信がない場合、文言等の軽微な調整は推奨案で
進めさせていただきますが、上記3点は仕様の根幹のため、ご返信を
いただいてからの着手とさせてください（スケジュールに影響する場合は
別途ご相談します）。
```

ポイントは3つ：

1. **推奨を先に出す**。ゼロから選ばせない（そこは元の設計と同じ）。ただし「どちらにしますか」ではなく「引っかかる点はあるか」を聞く。答えやすさの質が違う
2. **各項目に「崩れる条件」を付ける**。相手は選択じゃなくてリスク確認をするだけ。これが後の「聞いてなかった」を防ぐ
3. **無回答の扱いを明文化**。ただし仕様の根幹は無回答=承認にしない。ここを曖昧にすると手戻り責任がこっちに来る

会議で話さない人でも、メールで「引っかかる点だけ一言」なら返しやすいはず。返信が来たら、その内容を議事メモとして残しておくと、理解の記録にもなるよ。

## T3: 「CSVの列名一覧を表示するスクリプト」への回答

```python
#!/usr/bin/env python3
"""CSVファイルの列名一覧を表示する。

使い方: python show_columns.py data.csv
"""

import csv
import sys


def main() -> None:
    if len(sys.argv) != 2:
        print(f"使い方: python {sys.argv[0]} <CSVファイルのパス>", file=sys.stderr)
        sys.exit(1)

    path = sys.argv[1]
    try:
        with open(path, newline="", encoding="utf-8-sig") as f:
            reader = csv.reader(f)
            header = next(reader, None)
    except FileNotFoundError:
        print(f"ファイルが見つかりません: {path}", file=sys.stderr)
        sys.exit(1)

    if header is None:
        print("空のCSVファイルです", file=sys.stderr)
        sys.exit(1)

    for i, name in enumerate(header, 1):
        print(f"{i}: {name}")


if __name__ == "__main__":
    main()
```

`encoding="utf-8-sig"` にしてるのは、ExcelからエクスポートしたCSVの先頭に付くBOM（見えない文字）が1列目の名前に混ざるのを防ぐため。Shift_JISのCSVを扱うことが多いなら `encoding="cp932"` に変えてね。
